## ① 성능 지표 우선순위 — 왜 이 지표를 보는가

이 시스템의 핵심 로직은 **차량 탐지 → 5분 타이머 → 불법 주정차 판정 → AMR 출동 → 번호판 채증**이다. 각 단계에서 실패하면 단속이 끊기기 때문에 지표 선택이 중요하다.

| 순위 | 지표 | 왜 이 지표가 중요한가 | 이 시스템에서 실패하면? |
|---|---|---|---|
| ★★★ | val/cls_loss ↓ | 탐지된 bbox 안의 객체가 어떤 클래스인지 구분하는 능력. car↔id 혼동, 배경→객체 오인식 빈도에 직결. Recall/Precision과 별개로 "잡았는데 클래스가 틀리는" 오류를 잡는 유일한 지표 | 번호판을 배경으로 오인식 → 단속 증거 없음 / car를 id로 혼동 → 엉뚱한 객체 촬영 |
| ★★★ | id 클래스 AP ↑ | PR curve에서 번호판(id) 클래스만 따로 확인. 번호판을 얼마나 빠짐없이 잡는지 측정 | 번호판 못 잡음 → 재촬영 시도, 최대 3회 실패 시 수동 처리 요청 → 단속 실패 |
| ★★☆ | mAP@50-95 ↑ | bbox가 얼마나 정밀하게 그려지는지. IoU 0.5~0.95 구간 평균 → 엄격한 기준 | bbox가 느슨함 → OCR 입력 이미지 품질 저하 → 번호판 문자 인식 오류 증가 |
| ★★☆ | Recall ↑ | 탐지 자체를 놓치지 않는 비율. 차량을 못 잡으면 아예 단속 프로세스 시작 안 됨 | 차량 미탐지 → 출동 신호 없음 → 단속 0건 |
| ★☆☆ | Precision ↑ | 오탐(없는데 있다고 탐지) 비율. 낮으면 헛출동 증가. 단, AMR2가 현장 재확인하므로 완충 있음 | 오단속 3% 미만 요구사항 위반 위험. 단, 현장 재확인으로 대부분 걸러짐 |
| 필수 | 추론 FPS ≥ 10 | 요구사항 명시: 최소 10FPS 실시간 처리. YOLOv8n을 유지하는 이유 | 영상 끊김 → 탐지 누락 급증 / 실시간 단속 불가능 |

> ※ mAP@50은 참고용 — 종합 지표지만 bbox 정밀도는 mAP@50-95가 더 엄격하게 반영함

## ④ 실험 의도 & 결과 해석 — 왜 이 설정을 썼고 어떤 결론이 났나

### v1 — 기준 모델

- **실험 의도:** batch=16, 증강 없음. 이후 모든 실험의 비교 기준 설정
- **v1 대비 변경:** 없음 (기준)
- **결과:** cls_loss 0.398 / id AP 0.972 / mAP@50-95 0.7946
- **결론:** batch=16이 batch=32보다 전반적으로 우수함을 이후 실험(v2)을 통해 확인

---

### v2 — batch=32

- **실험 의도:** batch를 키우면 gradient 추정이 안정되어 학습 품질 향상 기대
- **v1 대비 변경:** `batch: 16 → 32` (나머지 동일)
- **결과:** Recall·id AP·cls_loss 모두 v1보다 낮음. FN도 더 많음
- **결론:** 데이터 302장 규모에서 batch=32는 오히려 역효과. 데이터가 적을수록 작은 batch가 각 샘플의 특징을 더 세밀하게 학습함. → **이후 모든 실험 batch=16 고정**

---

### v3 — 실험 A+C 🏆 최종 채택

- **실험 의도:**
  - 작은 객체(번호판)에 대한 대응력 강화 (copy_paste 증강)
  - lr 스케줄 안정화로 수렴 품질 향상 (lr0↓, lrf↑)
  - 조기 종료 방지로 후반 학습 유지 (patience↑, epochs↑)
- **v1 대비 변경:**
  - `lr0: 0.01 → 0.005` — 초기 학습률 낮춰 과적합 방지
  - `lrf: 0.01 → 0.1` — 최종 lr을 초기의 10%까지 유지, 후반 수렴 품질 개선 **(가장 영향 큰 변경)**
  - `warmup_epochs: 3 → 5` — 초기 불안정 구간 연장
  - `patience: 20 → 30` — 조기 종료 기준 완화
  - `epochs: 100 → 150` (실제 118ep에서 수렴)
  - `copy_paste: 0.0 → 0.3` — 번호판 객체를 다양한 위치에 합성하는 증강 추가
- **결과:** cls_loss **0.377 최저** ✅ / id AP **0.976 최고** ✅ / mAP@50-95 0.7793 (v1 대비 소폭 하락)
- **결론:** 핵심 2개 지표 동시 개선 성공. mAP@50-95 소폭 하락은 증강의 trade-off. 파라미터 튜닝 효과가 가장 잘 나온 조합. → **최종 채택**

---

### v4 — 실험 A+B+C

- **실험 의도:** id 클래스(번호판)에 더 집중하도록 cls loss 가중치 추가 (v3의 id FN=6 개선 목표)
- **v1 대비 변경:** v3 설정 전체 유지 + `cls: 0.5 → 2.5`
- **결과:** cls_loss 1.900 (스케일 달라짐 ⚠️) / id FN: 6 → 7로 오히려 증가
- **결론:** cls 가중치를 올리면 loss 숫자만 커질 뿐 실제 탐지 성능은 개선되지 않음. id FN 오히려 악화. → **cls 가중치 방향 완전 폐기**

---

### v5 — box=10

- **실험 의도:** v3의 약점(mAP@50-95 소폭 하락) 보완. box loss 가중치를 높여 bbox 정밀도 향상 기대
- **v1 대비 변경:** v3 설정 전체 유지 + `box: 7.5 → 10.0` / `copy_paste: 0.3 → 0.1`
- **결과:** mAP@50-95 **0.7513** (목표 0.779 미달 ❌) / cls_loss 0.395 (v3 0.377보다 높음 ❌) / box_loss 1.106으로 급등 ⚠️
- **결론:** box 가중치를 과도하게 높이면 bbox loss 자체가 불안정해짐. 두 목표 모두 미달. v3의 mAP@50-95 차이는 파라미터 튜닝으로 해결하기 어려움. → **box 가중치 방향 폐기 / 추가 데이터 수집이 더 효과적**

---

## ⑤ 최종 결론 및 다음 스텝

### 채택 모델

- **v3(A+C) best.pt 사용** — cls_loss 0.377 최저, id AP 0.976 최고. 번호판 탐지·오인식 모두 가장 우수
- **conf threshold → 0.43~0.45 즉시 적용** (재학습 없이 가능. 현재 0.87은 Recall을 스스로 낮추는 설정)

### 실험에서 배운 것

- batch=32는 데이터 302장 규모에서 역효과 → batch=16 고정
- `lrf=0.1` (0.01→0.1)이 학습 수렴 품질에 가장 큰 영향을 미쳤음
- cls 가중치(v4): loss 숫자만 커질 뿐 실제 성능 무관 → 폐기
- box 가중치 과도하게 상향(v5): bbox loss 불안정화 → 폐기
- mAP@50-95 격차(v3=0.779 vs v1=0.795)는 파라미터로 좁히기 어려움 → 데이터 추가가 더 효과적

### ⚠️ 미완료 항목

- **FPS 측정 미완료** → 10FPS 요구사항 충족 여부 반드시 확인 필요
